1. First Principle: The Web is More Than Text and Images

  • A webpage should deliver any kind of content, not just text and pictures.
  • Historically, embedding video/audio required third-party plugins (Adobe Flash, QuickTime, Silverlight).
  • Problems with plugins:
    • Inefficient
    • Insecure
    • Inconsistent across devices

Core Problem:

üëâ How can we embed video/audio in a standardized, native, plugin-free way that works across all browsers and devices?


Logical Solution:

üëâ HTML5 introduced the <video> and <audio> tags, designed specifically for embedding media.

2. <video> Tag Basics

Example 1: Minimal Usage

                    <video src="my-awesome-video.mp4"></video>
  • Only displays the first frame (static image).
  • No play/pause, volume, or controls.

Adding Controls

                    <video src="my-awesome-video.mp4" controls></video>
  • controls is a boolean attribute (no need forcontrols="true").
  • Enables built-in browser controls: play/pause, timeline, volume, fullscreen.

3. Intermediate Control with Attributes

  • width / height ‚Üí sets player dimensions

4.<audio> Tag

  • Works like <video> but only plays sound (no visuals).

    Example:

                    <audio src="audio/theme-song.mp3" controls></audio>
  • Features: play/pause, timeline, volume
  • Supports attributes: autoplay, loop, muted

5. Advanced: Real-World Solutions

Problem 1: Browser Format Support

  • Different browsers prefer different formats:
    • Chrome: .webm
    • Safari (iPhone): .mp4
    • Older browsers: .ogg
  • If only one format is provided, some users may not see the video.

Solution: <source> element

                    <video controls poster="images/thumbnail.jpg">
                        <source src="videos/my-video.webm" type="video/webm">
                        <source src="videos/my-video.mp4" type="video/mp4">
                        Sorry, your browser doesn't support embedded videos.
                    </video>
  • Browser picks the first supported format.
  • type helps avoid wasted downloads.

Problem 2: Accessibility

  • Deaf/hard of hearing ‚Üí need captions
  • Noisy/quiet environments ‚Üí need text
  • Different languages ‚Üí need subtitles

Solution: <track> element (WebVTT format)

                    <video controls>
                        <source src="video.webm" type="video/webm">
                        <source src="video.mp4" type="video/mp4">

                        <track src="captions_en.vtt" kind="captions" srclang="en" label="English">
                        <track src="subtitles_es.vtt" kind="subtitles" srclang="es" label="Espa√±ol">
                    </video>

Attributes:

  • src ‚Üí path to .vtt file
  • kind ‚Üí captions / subtitles / descriptions
  • srclang ‚Üí language code (e.g., en, es)
  • label ‚Üí name in video player menu

6. Embedding Media: Two Main Methods

Example 1: Robust & Professional Method (using <source>)

                    <video width="320" height="240" controls>
                        <source src="movie.mp4" type="video/mp4">
                        <source src="movie.ogg" type="video/ogg">
                        Your browser does not support the video tag.
                    </video>
  • Ensures maximum browser compatibility.
  • File format support:
    • .mp4 (H.264 codec) ‚Üí most widely supported
    • .webm (VP8/VP9 codec) ‚Üí open-source, supported in Chrome/Firefox
    • .ogg (Theora codec) ‚Üí older alternative

Logic:

  1. Browser checks the first <source>
  2. If supported ‚Üí plays it
  3. If not ‚Üí moves to the next option
  4. Fallback text shows if <video> isn‚Äôt supported at all

Pros: Reliable, professional, multiple fallbacks

Cons: Requires multiple encoded video formats

Example 2: Simple & Direct Method (using src)

                    <video src="a.mp4" controls></video>
  • Gives only one format option.
  • If unsupported ‚Üí video fails completely.

Pros: Very simple, quick setup

Cons: No fallback
